# Databricks x Dify 連携パターン — サマリ

## 課題

AIアプリ（Dify）が部門ごとに乱立し、LLMモデル・ツール・ナレッジが各アプリに分散。
誰が何を作ったか把握できず、コスト管理もアクセス制御も効かない状態。

## 解決策

AIアプリの「部品」をDatabricksに集約し、Difyは軽量なUIレイヤーに徹する。

- **LLMモデル** → AI Gatewayで一元管理・コスト制御
- **ツール** → UC Functionsで共有・権限管理
- **ナレッジ** → Vector Searchで一元管理
- **品質管理** → MLflow Tracingで自動評価

## 6つの連携パターン

| # | パターン | 概要 | ビジネス効果 |
|---|---------|------|------------|
| ① | LLMモデル連携 | DatabricksのAI GatewayでLLMを一括管理 | モデルコストの可視化・最適化 |
| ② | HTTP API連携 | 社内システム・DB機能をAPIで直接呼出 | 既存業務データとAIの即時連携 |
| ③ | MCP連携 | 標準プロトコルでツールを自動公開 | 設定だけでAIがツールを使える |
| ④ | RAG連携 | 社内ドキュメントをAIの知識として活用 | ナレッジの一元管理・鮮度維持 |
| ⑤ | バッチ処理 | Databricksから大量データをAI処理 | 定型業務の大規模自動化 |
| ⑥ | 観測性/品質管理 | AI回答の品質を自動評価・モニタリング | ガバナンスと品質の両立 |

## 推奨アーキテクチャ

```
┌─ Databricks（ガバナンス基盤）──────────┐
│  AI Gateway / UC Functions / Vector Search  │
│  MLflow Tracing（品質評価）                  │
└──────────────┬─────────────────────┘
               │ API / MCP
        ┌──────┼──────┐
       Dify   Dify   Databricks Apps
      (軽量)  (軽量)  (厳格な権限制御)
```

### 想定導入ロードマップ

| Phase | 期間 | やること | 狙い |
|-------|------|---------|------|
| **Phase 1** | 1-2ヶ月 | LLM + MCP + トレーシング | 最小工数でガバナンス効果を実感 |
| **Phase 2** | 2-4ヶ月 | HTTP API拡張 + 認証強化 + AI Judge | ツール・ナレッジの本格統合 |
| **Phase 3** | 4-6ヶ月 | バッチ処理 + Databricks Apps | 全社展開・厳格な権限制御 |

## 効果

**定量的メリット**
- LLMコストの一元可視化・最適化（AI Gateway）
- アプリ数が増えてもツール・ナレッジの重複ゼロ（UC Functions / Vector Search）
- AI回答品質の自動評価による運用コスト削減（AI Judge）

**定性的メリット**
- 「誰が・何を・どのデータで」AIを使っているかを把握可能
- アプリが乱立しても、データアクセス権限はDatabricks側で統制
- コードAgentとDifyの共存 — 部品を共有し、UIだけ使い分け
